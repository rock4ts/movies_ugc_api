# UGC API — Online Movie Theater

Flask service for the online movie theater platform. It accepts user activity events over HTTP, validates them, and publishes them to Kafka for downstream analytics and processing.

## What it does

The service is the **event ingestion layer** for user-generated content (UGC) analytics. Frontend clients send behavioral events — clicks, page views, movie interactions, search filters — either as authenticated users (via the platform **auth API**) or as anonymous users before sign-in.

**Validation** checks the request body against typed Pydantic models. Each event type has a discriminated `payload` shape; invalid UUIDs, timestamps, or missing fields are rejected before anything reaches Kafka.

**Authenticated ingestion** requires a valid RS256 access JWT in the `Authorization: Bearer <token>` header. The token is verified with the auth service public key. The `user_id` in the event body must match the JWT `sub` claim — this prevents one user from submitting events on behalf of another.

**Anonymous ingestion** accepts a client-generated `anonymous_id` and does not require a JWT. Anonymous events are published to a separate Kafka topic to keep identities and downstream processing isolated.

**Publishing** enqueues validated events to Kafka. Authenticated messages use `user_id` as the partition key and are published to `KAFKA_TOPIC`; anonymous messages use `anonymous_id` and are published to `KAFKA_ANONYMOUS_TOPIC`.

## Request flow

```text
Authenticated client
  │  POST /api/v1/events
  │  Authorization: Bearer <access_token>
  ▼
UGC API
  ├─ verify JWT signature and claims
  ├─ validate event schema
  ├─ ensure user_id == token.sub
  └─ produce to KAFKA_TOPIC (key=user_id)

Anonymous client
  │  POST /api/v1/anonymous-events
  ▼
UGC API
  ├─ validate event schema
  └─ produce to KAFKA_ANONYMOUS_TOPIC (key=anonymous_id)
```

## Supported event types

| `event_type` | Description |
|--------------|-------------|
| `click` | User clicked an interface element (`element_id`, `element_type`) |
| `page_view` | User opened a page or screen (`page`) |
| `movie_quality_changed` | User changed playback quality (`movie_id`, `previous_quality`, `new_quality`) |
| `movie_completed` | User finished watching a movie (`movie_id`) |
| `search_filter_used` | User applied search filters (`filters`) |

Authenticated event envelope:

| Field | Type | Description |
|-------|------|-------------|
| `event_id` | UUID | Unique event identifier |
| `user_id` | UUID | Authenticated user; must match JWT `sub` |
| `timestamp` | ISO-8601 datetime | When the event occurred |
| `payload` | object | Type-specific fields; `event_type` is the discriminator |

Anonymous event envelope:

| Field | Type | Description |
|-------|------|-------------|
| `event_id` | UUID | Unique event identifier |
| `anonymous_id` | UUID | Client-generated anonymous identity |
| `timestamp` | ISO-8601 datetime | When the event occurred |
| `payload` | object | Type-specific fields; `event_type` is the discriminator |

Authenticated example:

```json
{
  "event_id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "user_id": "7c9e6679-7425-40de-944b-e07fc1f90ae7",
  "timestamp": "2026-07-14T12:00:00+00:00",
  "payload": {
    "event_type": "click",
    "element_id": "btn_watch",
    "element_type": "button"
  }
}
```

## REST API

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `POST` | `/api/v1/events` | Bearer JWT | Ingest one user activity event |
| `POST` | `/api/v1/anonymous-events` | None | Ingest one anonymous activity event |

### Responses

| Status | Meaning |
|--------|---------|
| `202 Accepted` | Event validated and queued for Kafka delivery |
| `400 Bad Request` | Malformed JSON body |
| `401 Unauthorized` | Missing, invalid, or expired access token (authenticated endpoint only) |
| `403 Forbidden` | Token `sub` does not match event `user_id` (authenticated endpoint only) |
| `422 Unprocessable Entity` | Schema validation failed |
| `503 Service Unavailable` | Kafka producer unavailable or queue full |

Successful response:

```json
{
  "status": "accepted",
  "event_id": "3fa85f64-5717-4562-b3fc-2c963f66afa6"
}
```

### API documentation

Interactive Swagger UI is available at [`/openapi/swagger`](http://localhost:8000/openapi/swagger) when the API is running directly. The raw OpenAPI 3.1 spec is at `/openapi/openapi.json`.

Behind the platform nginx reverse proxy, docs are exposed at `/ugc/api/docs/` (for example, `http://localhost/ugc/api/docs/swagger`).

Documentation is generated by [flask-openapi3](https://github.com/luolingchun/flask-openapi3) from the route and model definitions. Use the **Authorize** button in Swagger UI to set a `Bearer` access token before trying `POST /api/v1/events`. `POST /api/v1/anonymous-events` does not require authorization.

Set `PUBLIC_URL_PREFIX` when the API is served behind a reverse proxy so Swagger UI "Try it out" requests use the public path. For example, with `PUBLIC_URL_PREFIX=/ugc`, requests go to `/ugc/api/v1/...` instead of `/api/v1/...`. Leave it empty for local development on port `8000`.

## Authentication

Access tokens are issued by the platform **auth API** and verified locally with its RS256 public key. UGC API does not issue or refresh tokens — it only validates them.

Required JWT claims:

| Claim | Description |
|-------|-------------|
| `sub` | User UUID; must equal the event `user_id` |
| `type` | Must be `"access"` |
| `exp` / `iat` | Standard expiry and issued-at timestamps |

Other claims (`role`, `access_labels`, `is_superuser`) are present in the token but not used for ingestion authorization — any authenticated user may submit their own events.

## Tech stack

- Python 3.12, Flask, [flask-openapi3](https://github.com/luolingchun/flask-openapi3) with Swagger UI
- Pydantic v2 for request/response schemas
- [confluent-kafka](https://github.com/confluentinc/confluent-kafka-python) producer (librdkafka)
- RS256 JWT verification (`PyJWT`, PEM public key from auth API)
- Gunicorn with 4 sync workers (Docker/production), or Flask dev server locally
- orjson for fast Kafka message serialization

## Environment variables

Copy `.env.example` to `.env` and adjust values for your environment.

| Variable | Description |
|----------|-------------|
| `FLASK_ENV` | `development` or `production` |
| `DEBUG` | Enable Flask debug mode |
| `KAFKA_BOOTSTRAP_SERVERS` | Kafka broker addresses |
| `KAFKA_TOPIC` | Target topic for authenticated events |
| `KAFKA_ANONYMOUS_TOPIC` | Target topic for anonymous events |
| `KAFKA_ACKS` | Producer acknowledgement level (default `all`) |
| `KAFKA_ENABLE_IDEMPOTENCE` | Enable idempotent producer (default `true`) |
| `KAFKA_LINGER_MS` | Producer batching delay in milliseconds |
| `KAFKA_BATCH_SIZE` | Producer batch size in bytes |
| `PUBLIC_KEY_PATH` | Path to auth API RS256 public key PEM file |
| `PUBLIC_URL_PREFIX` | Public URL prefix added to OpenAPI `servers` for Swagger "Try it out" behind a reverse proxy (for example, `/ugc`). Leave empty for direct local access on port `8000` |

In production, mount the auth service public key and point `PUBLIC_KEY_PATH` at it (for example, `auth-certs/jwt-public.pem`). In Docker Compose, set `PUBLIC_URL_PREFIX=/ugc` in `env-files/.env.ugc-api` so Swagger requests match the nginx route `/ugc/api/v1/`.

## Getting started

1. Create a virtual environment and install dependencies:
   - `pip install -r requirements.txt`
2. Copy `.env.example` to `.env` and set Kafka and JWT settings.
3. Place the auth API public key at the path configured in `PUBLIC_KEY_PATH`.
4. Run locally:
   - `python -m app.main`
   - The API listens on `http://0.0.0.0:8000`
   - Swagger UI: `http://localhost:8000/openapi/swagger`
5. Or build and run with Docker:
   - `docker build -t ugc-api .`
   - `docker run --env-file .env -p 8000:8000 -v /path/to/jwt-public.pem:/opt/ugc_api/auth-certs/jwt-public.pem:ro ugc-api`
   - The image starts Gunicorn with 4 sync workers on port `8000`

Send a test event (replace `<token>` with a valid access JWT and align `user_id` with the token `sub`):

```bash
curl -X POST http://localhost:8000/api/v1/events \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "event_id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
    "user_id": "7c9e6679-7425-40de-944b-e07fc1f90ae7",
    "timestamp": "2026-07-14T12:00:00+00:00",
    "payload": {
      "event_type": "page_view",
      "page": "/movies/42"
    }
  }'
```

Send an anonymous test event (no token required):

```bash
curl -X POST http://localhost:8000/api/v1/anonymous-events \
  -H "Content-Type: application/json" \
  -d '{
    "event_id": "7d9e6679-7425-40de-944b-e07fc1f90ae7",
    "anonymous_id": "e7f4f6a2-22cf-42d2-8d00-bb349f956c5f",
    "timestamp": "2026-07-14T12:00:00+00:00",
    "payload": {
      "event_type": "page_view",

```

## End-to-end tests

Functional tests exercise the running API against a real Kafka broker. They POST events over HTTP, assert on API responses, and consume published messages from Kafka to verify end-to-end delivery.

### Prerequisites

- Docker and Docker Compose
- Python 3.12+ with a virtual environment
- JWT test key pair in `tests/docker/certs/` (included in the repo)

If you need to regenerate the test certificates:

- `tests/docker/certs/jwt-private.pem` — used by tests to mint access tokens
- `tests/docker/certs/jwt-public.pem` — mounted into the API container for verification

From `ugc_api/`:

```bash
mkdir -p tests/docker/certs
openssl genrsa -out tests/docker/certs/jwt-private.pem 2048
openssl rsa -in tests/docker/certs/jwt-private.pem -pubout -out tests/docker/certs/jwt-public.pem
```

### Run tests

1. Install dependencies:
   - `pip install -r requirements-dev.txt`
2. Start test infrastructure:
   - `docker compose -f docker-compose.tests.yml up --build -d`
3. Run tests:
   - `pytest tests/functional -q`
4. Stop infrastructure:
   - `docker compose -f docker-compose.tests.yml down -v`

`docker-compose.tests.yml` starts three services:

- **kafka** — Apache Kafka 4.1.0 exposed on `localhost:29093`
- **kafka-init** — creates `ugc-events-test` and `ugc-anonymous-events-test` topics (3 partitions each)
- **ugc-api** — the API container built from the project Dockerfile, exposed on `localhost:18000`, configured via `.env.tests`

Before each test session, both Kafka topics are cleared so tests start from an empty stream.

### Test layout

```
tests/functional/
├── conftest.py          # HTTP/Kafka fixtures, JWT helpers, event builders
├── settings.py          # configurable test settings
└── testunits/events/
    ├── test_events.py       # happy-path ingestion for all event types
    ├── test_anonymous_events.py
    ├── test_auth.py         # JWT authentication and user_id binding
    ├── test_validation.py         # invalid payload and malformed JSON rejection
    └── test_anonymous_validation.py
```

### Coverage

**Happy path** (`test_events.py`) — parametrized over all supported event types. Each case verifies that `POST /api/v1/events` returns `202 Accepted` and that the event is published to Kafka with the user ID as the message key and the full payload in the value.

**Anonymous happy path** (`test_anonymous_events.py`) — parametrized over all supported event types. Each case verifies that `POST /api/v1/anonymous-events` returns `202 Accepted` and publishes to the anonymous topic with `anonymous_id` as the message key.

**Authentication** (`test_auth.py`) — verifies the API:

- rejects requests without an `Authorization` header (`401`)
- rejects invalid or expired tokens (`401`)
- rejects requests where `user_id` does not match JWT `sub` (`403`)
- accepts valid tokens with matching `user_id` (`202`)

**Validation** (`test_validation.py`) — verifies the API rejects:

- unknown `event_type` values
- invalid UUIDs and timestamps
- incomplete payloads (missing required fields)
- malformed JSON bodies

**Anonymous validation** (`test_anonymous_validation.py`) — verifies anonymous ingestion rejects invalid `anonymous_id`, malformed envelope fields, unsupported payloads, and malformed JSON.
